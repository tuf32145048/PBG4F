# 教材追加検証チェックリスト

教材追加を完了報告する前に使います。コンテンツだけの追加でも、最低限 `npm run validate` は実行します。

## 作業前

- [ ] `git status --short --branch` を確認した。
- [ ] 追加する章の `order` と前提章が明確である。
- [ ] Lesson Briefを埋めた。
- [ ] 新しく必要な用語とTipsを先に洗い出した。
- [ ] 変更範囲がコンテンツ中心であり、UIやスキーマ変更が必要か判断した。
- [ ] 判断できない設計思想や説明順は `notes/questions/` に質問として切り出した。

## 教材チェック

- [ ] Lesson JSONの `id`、`slug`、`order` が一意である。
- [ ] `summary` が実用的な一文になっている。
- [ ] `objectives` が「できること」として観察可能である。
- [ ] `prerequisiteIds`、`termIds`、`tipIds` が存在するIDを参照している。
- [ ] Lesson Markdownの説明順と問題の並びが対応している。
- [ ] コード例が短く、未説明の構文を避けている。
- [ ] 章の最後に、練習へ進む前の確認観点がある。

## 問題チェック

- [ ] 各問題に `id`、`slug`、`lessonId`、`title`、`level`、`conceptIds` がある。
- [ ] 1問が5から10分程度で解ける粒度である。
- [ ] 新概念を一度に詰め込みすぎていない。
- [ ] `statement`、`inputFormat`、`outputFormat` が曖昧でない。
- [ ] サンプルの `input` と `output` の改行が期待表示と一致している。
- [ ] ヒントが、軽い助言から具体的な実装へ段階的に進んでいる。
- [ ] 解答コードが実行可能で、未説明のショートカットを避けている。
- [ ] `explanationTrace` が「読む、分ける、書く、確かめる」の流れを示している。
- [ ] `commonMistakes` が実際に起こりそうなミスになっている。
- [ ] `reviewChecklist` が学習者の最終確認として使える。

## 用語・Tipsチェック

- [ ] 反復して出る語彙は用語化した。
- [ ] `short` は短く、初見で読める。
- [ ] `detail` は本文を邪魔しない補足になっている。
- [ ] `later` は今は深入りしない背景に分離されている。
- [ ] `aliases` が意図しない自動リンクを増やしすぎない。
- [ ] Tipsは本文理解に必須ではなく、補足として独立している。

## 表示・データチェック

- [ ] 説明文の用語自動リンクが有用に見える。
- [ ] コードブロックとインラインコードが用語リンク化されていない。
- [ ] 既存Markdownリンクが通常リンクとして残っている。
- [ ] 問題のチェックリスト文がチェックボックスと一緒に読める。
- [ ] 学習者向け本文に、未承認の外部ジャッジ・コンテストサイト名がない。

## コマンド

`web/` で実行します。

```powershell
npm run validate
```

UI、スキーマ、挙動、ルーティング、進捗保存を変えた場合は次も実行します。

```powershell
npm run test
npm run lint
npm run build
```

## 完了報告

- [ ] 変更ファイルを列挙した。
- [ ] 実行コマンドと結果を列挙した。
- [ ] 追加した教材・問題・用語・Tipsの数を示した。
- [ ] 残る内容ギャップを示した。
- [ ] ブラウザ確認ができない場合は制限を明記した。
- [ ] `notes/TODO.md` に作業記録を残した。
